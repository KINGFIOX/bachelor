二、已完成工作 — 仿真平台与调试基础设施

验证动机：设计 CPU → 需验证正确性 → 仿真验证平台

工具链：Chisel —[CIRCT]→ Verilog —[Verilator]→ C++ 类

这个流程下：verilator编译出的C++类在时钟信号驱动下的行为，就代表了 Chisel 描述的模块的行为

状态机视角：处理器有 ISA 状态机（软件可见）与 physical 状态机（微架构）。软件程序员只能看到 ISA 状态。RISC-V 标准委员会维护的 Spike 实现了 ISA 状态机。

差分测试原理：每条指令执行后，状态迁移（PC / mem / GPR）。若 DUT 与 Spike 在每条指令后的状态完全一致，则可断言：在软件程序员视角下，CPU 是正确的。

因此：开发了仿真测试平台，集成 Verilator（DUT）、Spike（Golden Model）、DiffTest 逐指令比对。 同时为了方便的 Debug，该平台还集成了 Capstone 反汇编引擎。均以静态库的形式链接。

二、已完成工作 — 多周期处理器核心

FSM 驱动的多周期数据通路

状态说明（7 个状态）

  • IDLE：上电复位后的初始状态，下一周期无条件进入 ifu.valid.wait 开始取指
  • ifu.valid.wait：等待 IFU 返回有效指令。访存指令→mem.ready.wait；非访存且无译码异常→WB；IFU/CU 异常→exception
  • mem.ready.wait：等待存储器接口就绪（AXI ready），就绪后→memValid.wait
  • memValid.wait：等待存储器返回有效数据（AXI valid）。LSU 异常→exception；否则→WB
  • WB：写回阶段。完成后→ifu.ready.wait；检测到中断→exception
  • exception：更新 mcause、mepc 等 CSR，完成后→ifu.ready.wait
  • ifu.ready.wait：等待 IFU 就绪，就绪后→ifu.valid.wait

二、已完成工作 — SoC 整体架构

总线架构设计思路：采用业界通用 AXI4 总线协议，利用 rocket-chip 的 Diplomacy 框架搭建 SoC 层次结构。CPU 核心连接 AXI4 Crossbar，访存延迟位于执行关键路径上，故一级 Crossbar 仅连接 SDRAM 以缩短路径；MROM、SRAM 为高速设备，直接连 Crossbar2；PSRAM(QPI)、UART、GPIO、PS2、VGA、Flash 等低速设备采用 APB 协议，经 APBXbar 与 AXI4ToAPB 桥接至 Crossbar2。

二、已完成工作 — SDRAM 控制器重写思路

SDRAM AXI4 测试平台（ultraembedded 提供）

Fork 了 ultraembedded/core_sdram_axi4。选取该仓库的原因：其提供 SystemC 编写的 UVM，可进行随机测试，测试平台如图。为验证 outstanding、乱序等特性，对 SystemC UVM 仿真平台进行了简单重构。

重写步骤：① 先用 Chisel 实现 SDRAM 芯片仿真模型；② 原控制器为 Verilog，用 Chisel 完整重写；③ 实现位扩展与字扩展（非简单移植）；④ 使用 Chisel Queue 对接 AXI4 总线，充分利用 outstanding、乱序等特性。

重写后的 Chisel 版本流水化程度更高，性能优于原 Verilog 版本。

二、已完成工作 — 乱序微架构设计

前后端解耦的乱序微架构

实现了前后端解耦的微架构：前端负责取指、后端负责指令执行。后端采用了乱序流水：译码、寄存器重命名、分派、发射、执行、写回、（晚执行）、提交，其中每一级流水采用握手信号实现时序解耦。

寄存器重命名

采用值捕捉方式，实现简单。后续计划采用统一物理寄存器方案，发射队列占用面积更小。

分支预测

目前使用静态分支预测，实现简单。后续计划采用 GShare 或基于感知机的分支预测器。

前后端解耦

单独验证前后端的正确性：验证后端时，使用软件实现的前端，通过解释执行获取 NPC，取到正确指令供给后端。前端负责分支预测、取指，可参照 ICache 做法对分支预测进行差分测试。
